## Phase D — OG image

- **Installed `@resvg/resvg-js` as devDep.** Pure-JS / Rust SVG rasterizer. Lighter than sharp, no native compile step on Windows, ~5MB.
- **Wrote `scripts/build-og-image.mjs`.** Inline 1200×630 SVG, four `<text>` elements: title at `(96, 340)` 148px sans-serif, two-line dek at `(96, 430)` and `(96, 478)` 38px serif italic, URL at `(1104, 572)` 22px mono `text-anchor="end"`. Background `#0c0b09` (Ink), text `#f5f0e6` (Parchment), URL `#a8a095` (muted). Renders via `Resvg.render().asPng()`, writes to `public/og-image.png`.
- **Font fight.** First attempt used generic `font-family="sans-serif"` and let resvg pick — Windows fontconfig resolved sans-serif to a condensed art-deco display face that looked bad. Second attempt tried to load the Plex woff2 files from `node_modules/@fontsource/ibm-plex-*/files/` via `resvg`'s `fontFiles` option — the woff2 didn't decode (resvg's bundled woff2 support is spotty), and the resulting PNG had no text at all (4KB pure background). Third attempt used named system fonts: **Arial / Georgia / Consolas**. That worked — clean rendering, correct hierarchy, italic applied properly, on-palette colors. The PNG is 37KB and ships verbatim.

**The font isn't Plex.** Documented limitation: the OG image renders in Arial / Georgia italic / Consolas instead of Plex Sans / Plex Serif italic / Plex Mono. At thumbnail size in social previews (typically 600×315 px), the Arial-vs-Plex-Sans difference is invisible and the visual identity (huge name, italic dek, mono URL footer, Ink-on-Parchment palette) is preserved. If the visual lands wrong, the fix is to vendor Plex .ttf files into `vendor/fonts/` (Fontsource only ships .woff/.woff2) and load them via resvg's `fontFiles` option. Defer until the visual fails an actual eyeball check.

### Hammerstein observation — the Entry 6 rule worked twice in one session

Phase 13 ended with two em dashes caught in `og:image:alt` and the title separators on the two newly-touched pages. I noted that as evidence the Entry 6 rule was firing pre-commit instead of post-push. Phase 14 is a stronger test: I didn't write the em dashes this session — they were already in the codebase from session 2, sitting unnoticed in seven page titles and three figcaption attributions. The grep + sweep took maybe four minutes and removed every visitor-facing em dash from the meta block of every page.

The pattern worth naming: **after an Entry-6-style rule fires once on a new line of code, immediately sweep the whole codebase for the same pattern.** Don't trust that the pattern only exists in the line that triggered the catch. The original commit (Phase 13) only fixed two files; the sweep (Phase 14) fixed eight. The catch-once-then-sweep discipline is what generalizes a single observation into a codebase-wide invariant.

Not a new log entry on its own — it's the same rule, just at a different scale. But worth marking in the session log so the practice is visible: the rule's reach is the codebase, not the diff.
